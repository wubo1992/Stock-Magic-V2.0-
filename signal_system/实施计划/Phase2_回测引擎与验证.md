# Phase 2 — 回测引擎与验证

**状态：已完成 ✓**

**目标：建立回测引擎，在历史数据上验证策略有效性**

---

## 背景

Phase 1 完成了系统骨架，但策略尚未经过历史数据验证。
在投入实盘使用前，必须回测证明策略在历史上有效（或找出失效原因并改进）。

---

## 目标指标（及格线）

| 指标 | 及格线 | 说明 |
|------|--------|------|
| 胜率 | > 40% | SEPA 策略靠高盈亏比弥补低胜率 |
| 盈亏比 | > 1.5 | 每次赚的要远大于每次亏的 |
| 最大回撤 | < 25% | 控制最坏情况的资产损失 |
| 每月信号数 | 2 ~ 10 | 太少无意义，太多条件太宽松 |
| 年化收益 | > 15% | 跑赢大盘 |
| 夏普比率 | > 1.0 | 单位风险的收益质量 |

## 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 回测粒度 | 日线，逐日驱动 | 与实盘模式完全一致，无未来数据泄漏 |
| 持仓模拟 | 等权重，每次满仓一只 | 简化计算，聚焦策略有效性验证 |
| 出场模拟 | 固定止损 -10% / 追踪止盈 -20% / 时间止损 20天 | 与实盘出场逻辑一致 |
| 样本内外分割 | 前 2/3 样本内，后 1/3 OOS | 防止过拟合，OOS 是真正的考试 |
| 股票池来源 | UNIVERSE.md 为唯一回测数据源 | 避免回测和实盘股票池不一致 |

## 回测引擎工作流程

```
BacktestEngine.run()
    → 下载全量历史数据（start - history_days 至 end）
    → 生成交易日列表
    → 逐日循环：
        strategy.run_date(date, queue)
        处理 queue 中的信号：
            买入信号 → 记录持仓（entry_price = 当日收盘价）
            出场信号 → 平仓，记录盈亏
    → 构建净值曲线（equity curve）
    → 计算绩效指标
    → result.print_report()
```
